# k6-test — Performance Testing with k6

This module demonstrates an alternative approach to the **gatling-tests** module using **[k6](https://k6.io/)** — an open-source load testing tool by Grafana Labs.

While Gatling uses a Scala/Java DSL compiled to JVM bytecode, k6 uses a JavaScript-based API that runs on Go. Both achieve the same goal: simulating concurrent users against the math API to measure throughput, latency, and error rates.

## Prerequisites

- **[k6](https://grafana.com/docs/k6/latest/set-up/)** — install via Homebrew or the official installer:

  ```bash
  brew install k6    # macOS
  # or
  sudo apt install k6  # Linux
  ```

## Directory Structure

```
k6-test/
  scenarios/
    math_api.js          # k6 load test script (mirrors Gatling MathApiSimulation)
  data/
    numbers.csv          # Feeder data for the mixed-operations scenario
  reports/               # Generated output (created on first run)
    k6-summary.txt       # Text summary of test results
  README.md
```

## Equivalent Load Profiles

Both modules test the same HTTP server (`localhost:32108`) with the same three scenarios:

| Scenario | Gatling Load Profile | k6 Load Profile |
|---|---|---|
| **Add Numbers** | `rampUsersPerSec(1).to(10).during(30s)` + `constantUsersPerSec(10).during(60s)` | `stages: [{30s, target:10}, {60s, target:10}]` |
| **Multiply Numbers** | Same as Add | (shares the same stages) |
| **Mixed Operations** | `constantUsersPerSec(5).during(90s)` | `stages: [{15s, target:5}, {90s, target:5}]` |

## Assertions / Thresholds

| Gatling | k6 |
|---|---|
| `global().responseTime().max().lt(500)` | `http_req_duration: ["p(95)<500"]` |
| `global().successfulRequests().percent().gt(95)` | `http_req_failed: ["rate<0.05"]` |

## Running the Tests

### Via Makefile (recommended — starts/stops the server automatically)

```bash
make k6-test
```

### Manually

```bash
# 1. Start the HTTP server
make start-server

# 2. Run k6
cd k6-test && k6 run scenarios/math_api.js

# 3. Stop the server
make stop-server
```

### With a custom duration

```bash
k6 run --duration 2m scenarios/math_api.js
```

### With a custom vU count

```bash
k6 run --vus 20 scenarios/math_api.js
```

### Export results to Grafana Cloud / InfluxDB / JSON

```bash
# JSON output
k6 run --out json=reports/results.json scenarios/math_api.js

# InfluxDB (requires running InfluxDB locally)
k6 run --out influxdb=http://localhost:8086/k6 scenarios/math_api.js
```

## Key Differences: k6 vs Gatling

| Feature | Gatling | k6 |
|---|---|---|
| Language | Scala / Java DSL | JavaScript |
| Runtime | JVM | Go (standalone binary) |
| Installation | Gradle plugin + Maven deps | Single binary (`brew install k6`) |
| Feeder data | CSV via `FeederBuilder` | `open()` + manual parsing |
| HTML reports | Auto-generated by Gatling | None built-in; use `k6 report` or export |
| Extensibility | Java ecosystem | JS + [k6 modules](https://jslib.k6.io/) |
| CI-friendly | Requires JVM | Zero dependencies, trivial in CI |
| Cloud integration | Gatling Enterprise | Grafana Cloud / On-prem |

## Why k6?

- **Zero setup** — one binary, no JVM, no Gradle plugin
- **JavaScript** — familiar to frontend/full-stack engineers
- **CI-native** — runs in Docker, GitHub Actions, GitLab CI with no extra deps
- **Built-in metrics** — no external monitoring required for basic tests
- **Extensible** — JS modules, custom metrics, websockets, GraphQL, gRPC
